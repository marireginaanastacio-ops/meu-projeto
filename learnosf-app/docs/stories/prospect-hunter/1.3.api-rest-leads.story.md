## Story
**As a** sistema (dashboard frontend),
**I want** uma API REST completa para ler, filtrar, atualizar e deletar leads do JSON local,
**so that** o dashboard possa consumir e manipular os dados dos prospects.

## Acceptance Criteria
1. `GET /api/leads` retorna todos os leads com suporte a query params: `platform`, `status`, `search` (busca em nome e bio), `limit` (default 50), `offset` (default 0). Resposta: `{ success: true, data: [...], total: N, limit: 50, offset: 0 }`
2. `GET /api/leads/stats` retorna contagens: `{ total, byStatus: { Novo, Contatado, Respondeu, Convertido, Descartado }, byPlatform: { instagram, linkedin }, conversionRate }`. Deve ser registrado ANTES de `GET /api/leads/:id` para evitar conflito de rota
3. `GET /api/leads/:id` retorna lead específico ou `404` se não encontrado
4. `PATCH /api/leads/:id` atualiza apenas campos permitidos: `status`, `notes`. Retorna lead atualizado. Valida que `status` é um valor válido do enum
5. `DELETE /api/leads/:id` remove lead do JSON e retorna `{ success: true, data: { deleted: id } }`
6. Todos os endpoints retornam JSON com status HTTP correto: 200 (ok), 201 (created), 400 (bad request), 404 (not found), 500 (internal error)
7. Error middleware global captura erros não tratados e retorna resposta padronizada
8. `lead.service.js` implementa toda lógica de CRUD sem acesso direto ao `req`/`res`

## Dev Notes

### Contratos de API (Source: architecture/prospeccao-automatizada-architecture.md#4)

```
GET  /api/leads?platform=instagram&status=Novo&search=maria&limit=50&offset=0
GET  /api/leads/stats
GET  /api/leads/:id
PATCH /api/leads/:id  body: { status?, notes? }
DELETE /api/leads/:id
```

### Status Enum Válidos (Source: prd/prospeccao-automatizada-PRD.md#FR6)
`"Novo"`, `"Contatado"`, `"Respondeu"`, `"Convertido"`, `"Descartado"`

### Resposta Padrão (Source: architecture/prospeccao-automatizada-architecture.md#4)

```json
// Lista
{ "success": true, "data": [...], "total": 42, "limit": 50, "offset": 0 }

// Stats
{
  "success": true,
  "data": {
    "total": 100,
    "byStatus": { "Novo": 60, "Contatado": 20, "Respondeu": 10, "Convertido": 8, "Descartado": 2 },
    "byPlatform": { "instagram": 60, "linkedin": 40 },
    "conversionRate": 8
  }
}

// Erro 404
{ "success": false, "error": "Lead not found", "code": "NOT_FOUND" }

// Erro 400
{ "success": false, "error": "Invalid status value", "code": "VALIDATION_ERROR" }
```

### Arquitetura de Camadas (Source: architecture/prospeccao-automatizada-architecture.md#6)
- Routes: apenas mapeamento, sem lógica
- Controllers: recebem req/res, chamam services, formatam resposta
- Services: toda lógica, sem acesso a req/res
- Utils: funções puras sem estado

### AVISO CRÍTICO: Ordem de Rotas
`GET /api/leads/stats` DEVE ser registrado antes de `GET /api/leads/:id`.
Se `:id` vier primeiro, a palavra "stats" será interpretada como um ID e retornará 404.
